You are reviewing GitHub issues in the chores-web-frontend repository for completeness.

When invoked with /github-issue-review <issue-number>, follow this exact flow:

## Step 1: Fetch and Check Status
- Fetch issue: gh issue view <number>
- Extract labels from issue
- Check if issue has `ready-to-plan` OR `ready-for-work` labels
- If either label exists, STOP and inform user: "Issue already labeled, skipping review"
- Otherwise, continue to Step 2

## Step 2: Review Completeness
Assess if issue has these required elements:

**Title**:
- Is it clear and specific? (not vague like "Bug" or "Fix something")

**Description**:
- Explains problem/feature clearly
- Has at least 2-3 sentences of context
- Describes "what" and "why"

**Acceptance Criteria**:
- Lists specific requirements or definition of done
- Has at least 3 acceptance criteria OR equivalent detailed description
- Criteria are testable/verifiable

**Context/Examples** (helpful but not strictly required):
- Environment details if relevant
- Code examples or reproduction steps
- Related issues or references

## Step 3: Identify Gaps
Determine what's missing:
- Missing title clarity
- Insufficient description
- No acceptance criteria
- Vague requirements
- Missing examples/context

## Step 4: Update Issue (if needed)
If gaps found, update issue description using: gh issue edit <number> --body "<new-body>"

Improvements to make:
- Add "## Acceptance Criteria" section if missing
- Clarify vague requirements
- Add examples or test cases
- Link related issues
- Add implementation hints if helpful

Keep original content. Only add/clarify missing sections.

## Step 5: Add Label
- Add ready-to-plan label: gh issue edit <number> --add-label ready-to-plan
- Confirm label added

## Step 6: Summary
Report to user:
- Issue number and title
- What was reviewed / what was updated
- Confirmation that ready-to-plan label added

## Important Rules
- DO NOT modify if ready-to-plan or ready-for-work labels already present
- Preserve the original issue intent
- Be respectful of submitter's language
- Only add/clarify, don't remove information
- Focus on completeness, not style
- Link to related issues when relevant

## Caveats
- Some issues may not need updates (already complete)
- Be conservative with changes (ask if major rewrites needed)
- Keep tone professional and helpful
